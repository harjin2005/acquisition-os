# SOP System

**Purpose:** anything done twice gets a procedure; anything with legal/compliance weight gets a checklist. SOPs are files (`/capabilities/<name>/sops/`), owned, dated, and retired when unused.

**Format (normative):** name · owner · trigger ("run this when…") · steps (checkable) · tools/links · escalation · last-used · review-by.

**V1 seed set (each ≤1 page):** Weekly review · Monthly close · Quarterly OKR + board pack · Decision record (Type 1 / Type 2) · Approval object issuance · Meeting → memory · End-of-session write-back · New hire onboarding (day 1 = MASTER_CONTEXT + memory access) · Contract intake (Legal) · Vendor add/review · Incident response (severity ladder shared with product) · Privacy deletion (the only memory-deletion path) · Design-partner onboarding (mirrors DOC-121 weeks 1–4) · Research brief production · Content publish checklist (claim-verification gate) · Restore drill (points to product runbook) · New capability proposal (spec template + measurable-outcome requirement) · Startup Factory instantiation (StartupFactory.md §3, the master SOP).

**Automation ladder for SOPs themselves:** L0 checklist → L1 skill that walks it → L2 scheduled with approval object. Promotion only after the checklist has been executed manually ≥3 times (you can't automate what you haven't standardized).

**KPIs:** coverage of recurring work · SOPs used in last quarter (cull the rest) · checklist-skips discovered in incidents (target 0).
