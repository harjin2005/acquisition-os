# Operations Capability (incl. Product-ops & Customer Success seeds)

**Purpose:** the company's connective tissue — SOPs, tasks, hiring, vendors, planning — so nothing depends on the founder's working memory.

**Core responsibilities:** SOP system (SOPs.md) · task/epic hygiene (tracker synced to memory; stale-task sweeps) · hiring pipeline (role docs, structured interview kits, decision records for hires = Type 1) · vendor register (owner, cost, renewal date, data-access level — feeds Legal + Security) · planning rituals support (FounderOS cadence logistics).
**Product sub-capability (L1):** interview scheduling + note templates → insight extraction into memory (customer knowledge) → feedback clustering vs DOC-121 priorities → PRD drafting in house style with evidence links. Boundary: prioritization is proposed with evidence, decided by humans; the "not-build" list is enforced in review.
**Customer Success sub-capability (L0, pre-customers by design):** onboarding SOP matching DOC-121's scripted weeks 1–4 · health-score definition (from DOC-120 §10) implemented as a metrics query, ready to activate at first customer · support knowledge-base seeded from design-partner questions.
**Inputs:** tracker, calendar, interviews, vendor docs. **Outputs:** SOPs, role kits, insight reports, PRD drafts, vendor register. **Memory writes:** all of the above + hiring decision records.
**Approval rules:** hires, vendor contracts, SOP changes affecting compliance = founder sign-off; everything else L1 drafts.
**Failure modes:** SOP graveyard (written, never used) → each SOP has an owner + last-used date, quarterly cull; tracker/memory drift → daily sync job with diff alarm; interview insights biasing to loudest voice → clustering reports show n and source spread.
**KPIs:** % recurring work with a current SOP · task staleness · hiring cycle time · vendor renewals missed (target 0) · interview→insight latency.
